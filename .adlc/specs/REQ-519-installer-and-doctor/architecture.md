# Architecture — REQ-519: One-Command Installer and `adlc doctor`

## Goal

Two entry points that turn toolkit install from "undocumented machine surgery"
into two commands:

1. **`install.sh`** at the repo root — idempotent, repair-capable bootstrap:
   symlinks (`~/.claude/skills`, `~/.claude/agents`), PATH/shim wiring for the
   `adlc` CLI, config scaffold, optional (opt-in) delegation install, ending in
   an embedded `adlc doctor` run.
2. **`adlc doctor`** — read-only diagnostic that checks every environmental
   dependency end-to-end, prints pass/fail/skip per check with a copy-pasteable
   remediation, and is the pre-flight primitive other skills can call via
   `--checks`.

`adlc` ships as an umbrella CLI under `tools/adlc/` (BR-11); `doctor` is its
first subcommand and dispatch is data-driven so later REQs (`renumber`, tier
render) add commands without touching dispatch.

## Design decisions (ADRs)

### ADR-1: `adlc` is a Python CLI under `tools/adlc/`, mirroring `tools/kimi/`

The `tools/` carve-out (conventions.md) already exempts real executable code
from the markdown-only rule, and `tools/kimi/` is the proven pattern: a Python
module (`_common.py`) plus thin shell-script shim wrappers in `~/bin/` that
`exec` the module through the venv, generated by an `install.sh` with no
hardcoded HOME beyond `$HOME`. `adlc` follows the same shape:

```
tools/adlc/
├── adlc.py              # umbrella entry: data-driven subcommand dispatch
├── doctor.py           # the `doctor` subcommand: check registry + runner + report
├── checks.py           # individual check functions (one per Check id)
├── README.md
└── tests/
    ├── conftest.py      # repo-root path injection (mirrors tools/kimi/tests)
    ├── test_doctor.py   # check runner, verdict, --checks filter, exit code
    └── test_checks.py   # per-check pass/fail/skip logic with fixtures
```

`adlc` needs **no third-party deps** (pure stdlib: `os`, `sys`, `subprocess`,
`shutil`, `platform`, `argparse`, `json`) so it does NOT require the kimi venv —
the shim execs the system `python3` directly. This keeps `adlc doctor` runnable
on a machine that has NOT opted into delegation (BR-9, AC: "delegation checks =
skip/disabled"). Rationale mirrored, not just mechanism (LESSON-023): doctor
must run before/without delegation, so it cannot depend on the delegation venv.

### ADR-2: Data-driven check registry, not a switch

`doctor.py` holds an ordered registry: a list of `Check` descriptors, each with
`id`, a callable returning `(result, detail, remediation)`, and an
`applies_to(profile)` predicate for platform gating (BR-6). The runner iterates
the registry, applies the optional `--checks id1,id2` filter, and prints an
ordered report. Adding a check = appending one descriptor. The `--checks` filter
(BR-8, the skill pre-flight contract) is a set-membership filter over check ids,
validated against the registry so an unknown id is an error, not a silent no-op.

### ADR-3: `CheckResult` is a 3-state enum, exit code derived

`pass` / `fail` / `skip`. `skip` carries a reason (e.g. "launchctl is
macOS-only", "delegation not enabled"). Doctor exits **non-zero iff any non-skip
check fails** (BR-4) — `skip` never contributes to failure. This makes the
Linux-on-macOS-checks and not-opted-into-delegation cases pass the overall
verdict honestly (BR-6, AC-2).

### ADR-4: `delegate-gate` check reuses REQ-515, does NOT reinvent it

The `delegate-gate` check sources `partials/delegate-gate.sh` (shipped by
REQ-515) and interprets its **0/1/2 return-code contract** plus
`ADLC_DELEGATE_GATE_REASON`:

| gate rc | reason | doctor result |
|---|---|---|
| 0 (`ok`) | delegated path live | **pass** — "delegation enabled and reachable" |
| 1 (`not-opted-in` / `disabled-via-env`) | opt-in off | **skip** — reason carried through |
| 2 (`no-binary`) | `adlc-read` not on PATH | **skip** if delegation not opted-in; **fail** with remediation if config says `enabled: true` but binary missing (misconfigured) |

The "misconfigured" fail (config enabled but binary absent / config malformed)
is the case BR-4 names ("delegation gate state … misconfigured"). Config
resolution is read through the same surface REQ-515 ships — `tools/kimi/_common.py`
`parse_delegate_config()` for the config-file read — invoked via a subprocess
probe so `adlc` keeps zero hard dependency on the kimi module being importable
(it lives under a sibling `tools/` dir, may be absent on a skills-only install).
The Assumption ("REQ-515 config settled") is resolved: REQ-515 is merged into
this branch's base, so the check ships fully, NOT as `skip: not yet specified`.

### ADR-5: All file mutation goes through one atomic helper (`_atomic_write`)

Every `~/.zshrc` / settings.json / config-scaffold mutation in `install.sh` uses
backup → temp-write → `mv` rename (BR-2). Marker-guarded idempotent appends
(`# >>> adlc-toolkit (REQ-519) >>>` … `# <<< adlc-toolkit (REQ-519) <<<`) for
`~/.zshrc`, content-compared for symlinks (relink only if target differs), so a
second run is a true no-op (BR-1, AC-3). Malformed existing content (e.g. a
half-written marker block) produces an actionable message, never a traceback or
silent overwrite (BR-2). This mirrors `tools/kimi/install.sh`'s established
marker + atomic-settings pattern rather than inventing a new one.

### ADR-6: No hardcoded user paths — derive `REPO_ROOT` at runtime (BR-3)

`install.sh` resolves `REPO_ROOT` via `git -C "$(dirname "$0")" rev-parse
--show-toplevel` (with a script-relative fallback), exactly like
`tools/kimi/install.sh`. The generated `adlc` shim and config scaffold embed
only `$REPO_ROOT` and `$HOME` derived at install time. `install.sh --repair`
re-derives and regenerates everything, so moving the clone and re-running
repairs all symlinks/shims with the new path and leaves no stale absolute path
from the old location (BR-3, AC-4). Verified by the AC-4 grep.

### ADR-7: `--dry-run` is a global plan-only mode threaded through the mutators

`install.sh` parses `--dry-run` / `--repair` into a mode flag. In `dry-run`,
every mutator prints its intended action (`would symlink …`, `would append PATH
to ~/.zshrc`) and returns without touching the filesystem (BR / AC-7). The
plan is the same code path as the real run (the mutators branch at the write
boundary), so the plan cannot drift from the action.

## Check registry (the doctor surface, BR-4)

Ordered, each with a concrete pass/fail example (BR-5 — example lists, not prose):

| id | passes when | fails when (remediation) | skip when |
|---|---|---|---|
| `skills-symlink` | `~/.claude/skills` is a symlink resolving into a git checkout | missing/dangling/not-a-symlink → `ln -sfn <REPO_ROOT> ~/.claude/skills` | — |
| `agents-symlink` | `~/.claude/agents` resolves into the checkout | dangling → `ln -sfn <REPO_ROOT>/agents ~/.claude/agents` | — |
| `toolkit-clean` | clone is on a branch and not unexpectedly dirty | detached HEAD or dirty → `git -C <root> status` | — |
| `path-shims` | `adlc` resolves on PATH and `adlc --version` exec's | not on PATH → `./install.sh --repair` then restart shell | — |
| `gh-present` | `gh` on PATH | absent → install `gh` (per-OS line) | — |
| `gh-auth` | `gh auth status` exits 0 | unauth → `gh auth login` | skip if `gh` absent (chained on `gh-present`) |
| `git-identity` | `git config user.name` and `user.email` both set | unset → `git config --global user.email …` | — |
| `delegate-gate` | gate rc 0 | gate rc 2 **and** config `enabled: true` → fix config/PATH | gate rc 1, or rc 2 with delegation not opted-in |
| `counters` | each `~/.claude/.global-next-{req,bug,lesson}` present, numeric, and its `.lock.d` not stale | non-numeric/stale-lock → exact `printf`/`rmdir` fix | skip a counter file that legitimately doesn't exist yet (first-run) |
| `launchctl` | (macOS) kimi setenv LaunchAgent loaded **if** delegation opted-in | load failed → `launchctl bootstrap …` | skip on Linux (BR-6); skip if delegation not opted-in |
| `template-version` | project `.adlc/` scaffold not drifted from toolkit templates | drift → pointer to `/template-drift` | skip when run outside a consumer project (no project `.adlc/templates/`) |

`claude-code` (version/model) is **report-only**, never fails the verdict
(Open Question resolution). The permission-allowlist is **checked/reported** by
doctor, **written** by `/init` (Open Question resolution).

## Machine profile (BR-6)

`doctor.py` builds a `Profile` once: OS family via `platform.system()`
(`Darwin`/`Linux`), and the **real login shell** — read from the password
database (`pwd.getpwuid(os.getuid()).pw_shell`) NOT `$SHELL` (which reflects the
invoking shell, not the login shell — the BR-6 honesty requirement). macOS-only
checks (`launchctl`) carry `applies_to=lambda p: p.os == "Darwin"` and emit
`skip` with a notice on Linux; the symmetric case (a Linux-only check) would skip
on macOS. Skip, never fail, on inapplicable checks.

## Shell safety (BR-7)

`install.sh` is `#!/usr/bin/env bash`, `set -euo pipefail`, BSD-safe (no GNU-only
flags; `sed` writes to temp + rename, no in-place `-i ''` footgun;
`grep -F` for marker detection). Dogfooded under `zsh -c` and `bash -c` on macOS
and `bash` on Linux. The Python (`adlc`) is cross-platform stdlib. Any fenced
blocks added to skills (README is not a skill) follow lint-skills conventions —
not applicable here since we add no SKILL.md fenced shell.

## Skill pre-flight convergence (BR-8)

The `--checks gh-auth,delegate-gate` filter (AC-6) is the contract that lets
`/sprint` and `/proceed` call `adlc doctor --checks <subset>` instead of
maintaining parallel probe code. This REQ ships the **filter mechanism and the
checks**; it does NOT rewrite the sibling skills' preflight in this REQ beyond
documenting the contract in `tools/adlc/README.md` and auditing that no
duplicated probe shell exists (audit finding: sibling skills currently describe
probes in prose, not concrete shell — so there is no duplicated executable probe
to delete; the convergence is forward-looking and the contract is now available).

## README (BR-10)

The README "Install" section is rewritten to exactly two commands
(`git clone …` + `./install.sh`), with everything else ("did it work?",
remediation, delegation opt-in) pointed at `adlc doctor`. The detailed manual
symlink steps move to an "under the hood / manual install" appendix so the
knowledge isn't lost but the happy path is two commands.

## Task graph

```
TASK-001 (adlc CLI skeleton + dispatch)   ─┐
                                            ├─→ TASK-003 (doctor checks)  ─→ TASK-004 (install.sh)  ─→ TASK-006 (README + docs)
TASK-002 (doctor runner + profile + report)┘            │                          │
                                                         └─→ TASK-005 (tests) ──────┘
```

- **TASK-001** — `tools/adlc/adlc.py` umbrella entry + data-driven subcommand dispatch + `adlc.py doctor` wiring; `--version`.
- **TASK-002** — `doctor.py`: `Profile` builder (OS + real login shell), `CheckResult` enum, `Check` descriptor, registry runner, `--checks` filter + unknown-id error, ordered report formatter, exit-code derivation.
- **TASK-003** — `checks.py`: implement all checks in the registry table (symlinks, toolkit-clean, path-shims, gh-present/auth, git-identity, delegate-gate via partial, counters, launchctl, template-version), each returning `(result, detail, remediation)`.
- **TASK-004** — root `install.sh`: arg parse (`--repair`/`--dry-run`), `REPO_ROOT` derivation, `_atomic_write` helper, symlink/PATH/shim/config-scaffold mutators (idempotent, marker-guarded), opt-in delegation step (defers to `tools/kimi/install.sh`), embedded final `adlc doctor` run.
- **TASK-005** — `tools/adlc/tests/`: pytest for dispatch, doctor runner, `--checks` filter, exit codes, per-check pass/fail/skip with fixtures; mirror `tools/kimi/tests` conftest path-injection.
- **TASK-006** — README install rewrite (BR-10), `tools/adlc/README.md`, and the BR-8 sibling-skill audit note.

Dependency order (sequential in subagent mode): 001 → 002 → 003 → 004 → 005 → 006.
(002 depends on 001's module layout; 003 depends on 002's Check/CheckResult types;
004 generates the shim that runs 001; 005 tests 001–003; 006 documents the whole.)

## Test strategy

- `adlc` checks are pure functions over an injected environment (profile + a
  small filesystem/PATH probe surface), so `test_checks.py` drives them with
  `tmp_path` fixtures (fake `~/.claude` trees, fake counters, monkeypatched
  PATH) to exercise pass/fail/skip without touching the real machine — the
  same offline-test discipline as `tools/kimi/tests`.
- `test_doctor.py` covers the runner: registry iteration order, `--checks`
  subset, unknown-id rejection, verdict/exit-code derivation (skip never fails).
- `install.sh` idempotency/dry-run are validated by the ACs via dogfooding
  (before/after filesystem diff) — documented in `tools/adlc/README.md` test
  notes; the shell is kept thin and delegating so the testable logic lives in
  Python.
